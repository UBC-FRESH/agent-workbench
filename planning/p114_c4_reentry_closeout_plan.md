# P114 C4 Re-entry and Closeout Plan

## Purpose

Close P114 as the capability-validity prerequisite for the parked P107 C4+
configuration suite. P114 does not run the comparative suite itself and does
not make an economics claim. It determines whether the frozen C4 Worker route
can enter that suite without a transport or tool-surface defect contaminating
the result.

## Accepted route and excluded route

The accepted route is a fresh Codex CLI controller rooted in the literal P114
worktree. It stages the role-bound package MCP server through the reversible
transaction, spawns exactly one `ollama_qwen_coder_worker` with
`fork_context:false`, and supplies the generated ticket verbatim. The Worker
uses deferred discovery and child-native namespaced package `exec` and
`apply_patch` calls.

`p114_package_mcp_composite_r1` accepted this route for the bounded
read-to-patch-to-validate sequence. The current VS Code nested-host custom-tool
route remains useful negative integration evidence, but it is not the route
that P114 will qualify or use to unblock P107. Do not reopen that route during
P114 closeout unless the product supplies a concrete nested-host dispatch
repair.

## Remaining P114 tasks

### P114.3 — normalize the deployment decision

- Synchronize the package tool matrix, P114 plan, roadmap, and parent issue
  with the already accepted fresh CLI-parent package `exec`, `apply_patch`,
  deferred-discovery, and composite proofs.
- Record the VS Code nested-host route as excluded from the P107 entry route,
  rather than silently treating its known dispatch gap as a Qwen failure.
- Rerun focused package, adapter, fixture, transaction, and policy checks from
  clean processes.

### P114.4 — freeze and battery-test the P107-admissible capability bundle

- Freeze one exact role profile, provider/catalog configuration, worktree,
  package server command, grant file, ticket template, deterministic target,
  and evidence manifest for the C4 entry route.
- Add one deterministic admission validator that rejects missing deferred
  discovery, a non-namespaced package call, an ungranted command or patch, any
  fallback tool, incorrect target content, missing policy/outcome records, or
  non-byte-for-byte restoration.
- This is a capability bundle, not a new general toolset and not an economics
  measurement.
- Run **three independent fresh package-route composite observations** using
  that exact bundle. One accepted composite proves the primitive route; it does
  not replace the preregistered three-session battery. Every row must pass the
  admission validator.

## P114 closeout gate

P114 can close when the completed three-row package battery and two baseline
qualification observations are accepted; the P107 re-entry packet, roadmap,
change log, parent issue, and public-safe plan agree on the qualified route
and exclusions; and the P114 branch has merged and #661 has closed. Then—and
only then—P107 may activate its baseline C4 suite. Each C4+ profile must first
pass its own non-comparative package admission observation before it can enter
the comparable suite or the separate economics boundary required for a P107
decision.
